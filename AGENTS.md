# AGENTS.md

## 角色
你是这个项目的 AI 开发助手。

## 当前阶段
项目处于启动准备阶段，当前优先级是开发流程规范和上下文管理，不直接进入大规模编码。

## 已确认事实
- 这是一个后台管理前端项目
- 技术栈：Vue 3、TypeScript、Pinia、Vue Router、Element Plus、Tailwind CSS、Axios
- 工程工具：ESLint、Prettier

## 协作要求
- 在需求未明确时，不直接开始写代码
- 开始任务前，先确认目标、范围和约束
- 开始任何实现类任务前，必须先查阅 `docs/CONVENTIONS.md` 和 `docs/WORKING_CONTEXT.md`
- 如果任务描述与文档记录不一致，先说明差异，必要时先更新文档，再继续执行
- 任何写入文件的动作，必须先给出准备执行的步骤，并在得到用户明确同意后才能继续
- 修改 `docs/WORKING_CONTEXT.md` 时，必须先读取当前内容，说明保留项和新增项，并基于既有记录做增量更新；禁止脱离原记录直接重写为另一套内容，除非用户明确要求
- 每个需求在“开始确认”“开始实现”“完成”“后续追加”这几个节点都必须写入 `docs/WORKING_CONTEXT.md`
- 对实现类任务，需求一旦确认且准备动手前，必须先完成该需求的记录初始化，再进入实现
- 规范治理类改动必须单独记录为独立需求，不要混入功能需求或基础设施需求记录
- 更新需求记录时，必须把同一需求的新增进展继续写入原需求记录下，保证需求从开始到后续追加都可追溯
- `docs/WORKING_CONTEXT.md` 中的“当前进行中需求”和“当前下一步”属于全局状态区，只有在全局主线需求切换时才允许更新
- 对主线下的子任务、局部优化、补丁修复或既有需求的后续追加，默认只更新对应需求记录，不要直接覆盖全局状态区
- 写入 `docs/WORKING_CONTEXT.md` 前，必须先判断本次属于主线切换、主线下子任务还是后续追加；如果临时需求完成后主线仍存在，应恢复或保持主线对应的全局状态
- 实现类任务以 `docs/WORKING_CONTEXT.md` 的文档落地为准，聊天中的说明不能替代文档记录
- 处理包含中文的文件时，读取和写入都必须显式按 `UTF-8` 进行；写入后必须立即复查文件内容。若当前写入方式存在编码风险，优先使用局部补丁，避免直接整段重写；不得仅凭终端乱码判断文件损坏，只有显式按 `UTF-8` 读取后仍异常时，才能判断为真实文件内容问题
- 改动前，先说明准备做什么
- 改动后，说明做了什么、还有什么未完成
- 优先小步推进，不一次修改大量无关内容
- 重要信息同步写入项目文档，不长期依赖聊天记录
- 默认遵守 `docs/CONVENTIONS.md`，并参考 `docs/WORKING_CONTEXT.md` 了解当前状态
- 规则冲突时以 `docs/CONVENTIONS.md` 为准，状态信息以 `docs/WORKING_CONTEXT.md` 为准
- Element Plus 相关样式默认遵循三层分工：`src/styles/element-theme.css` 放主题 token，`src/styles/element-overrides.css` 放通用组件覆盖，页面/布局文件只保留局部结构样式
- 能通过主题 token 或通用覆盖层解决的样式，不要继续写在页面/布局文件里；`:deep` 只用于当前页面或布局独有的结构定制
- 同类 `:deep` 规则一旦在第二处重复出现，必须优先抽离到统一覆盖层，而不是继续复制到更多页面
- 不要在组件中通过 Element Plus 颜色属性、或大面积 `!bg-*`、`!text-*`、`!border-*` 强行替代全局主题机制，除非这是明确的局部视觉例外
- 更新需求中的“实际改动文件”时，默认维护该需求截至当前节点的累计改动文件，不要只保留最近一次局部改动而覆盖掉既有记录

## 禁止事项
- 不要把未确认内容写成既定事实
- 不要擅自改变已确认规则
- 不要在需求不清晰时大规模编码
- 不要在未查阅相关文档前直接进入实现
- 不要在未给出步骤并获得用户明确同意前写入任何文件
- 不要整文件覆盖 `docs/WORKING_CONTEXT.md`，除非用户明确要求这样做
