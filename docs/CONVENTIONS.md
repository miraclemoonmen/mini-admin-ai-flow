# CONVENTIONS.md

## 目的
这个文件用于约定项目的开发流程、记录方式和日常执行规则。

## 任务流程
每次任务默认按以下顺序执行：
1. 确认任务目标
2. 确认任务范围
3. 查阅已有文档
4. 说明准备执行的步骤
5. 获得用户明确同意
6. 补充必要上下文
7. 开始实现或写入文件
8. 完成后更新记录

## 执行规则
- 需求还在讨论时，先整理问题和范围，不直接进入编码
- 任务明确后，先补当前任务背景，再开始修改
- 优先小步推进，不一次改很多无关内容
- 如果实现和原计划不一致，需要补充原因
- 提交前完成基础检查和格式整理
- 不提交明显未完成的临时代码
- 未经用户明确确认的技术选择、工具选择和初始化命令，不得写入既定事实
- 任何写入文件的动作，必须先说明步骤，并在得到用户明确同意后才能执行
- 修改 `docs/WORKING_CONTEXT.md` 时，必须先读取当前文件内容，说明保留项和新增项，保留未被明确要求变更的既有事实和记录，再在对应位置追加或局部更新；禁止整文件重写，除非用户明确要求
- 处理包含中文的文件时，读取和写入都必须显式按 `UTF-8` 进行；写入后必须立即复读校验。若当前写入方式存在编码风险，优先使用局部补丁，避免直接整段重写；不得仅凭终端乱码判断文件损坏，只有显式按 `UTF-8` 读取后仍异常时，才能认定为真实文件内容问题

## 文档规则
项目当前维护以下文档：
- `AGENTS.md`：AI 助手协作规则
- `docs/CONVENTIONS.md`：项目开发流程和执行规范
- `docs/WORKING_CONTEXT.md`：当前项目状态、已确认事项和需求滚动记录

文档更新方式：
- `AGENTS.md` 采用覆盖式更新，保持最新协作规则
- `docs/CONVENTIONS.md` 采用覆盖式更新，保持最新项目规范
- `docs/WORKING_CONTEXT.md` 采用滚动更新，保留固定结构并持续刷新当前状态与需求记录，不作为纯追加日志使用
- `docs/WORKING_CONTEXT.md` 的固定结构指一级章节框架应保持稳定；需求进展以需求记录形式在固定章节内滚动更新
- `当前阶段目标` 只记录阶段级目标，不重复展开某个具体需求的完整进展；具体需求进展统一写入 `需求记录`
- 规范治理类改动应独立建需求记录，不与功能需求或基础设施需求混写
- `当前下一步` 是全局唯一下一步来源，只记录当前进行中需求的下一步，并且每条必须带需求编号
- `当前进行中需求` 和 `当前下一步` 属于全局状态区，只有在全局主线需求发生切换时才更新
- 非主线的子任务、局部优化、补丁修复或已完成需求的后续追加，默认只更新对应需求记录，不覆盖全局状态区
- 如果临时插入的独立需求完成后，项目仍有既定主线需求在持续进行，应将全局状态区保持或恢复到该主线需求

## 上下文规则
以下内容必须写入文档：
- 已确认的项目事实
- 当前阶段目标
- 当前进行中需求
- 需求记录
- 关键决定
- 阻塞项
- 当前下一步

## 任务记录要求
当前阶段不单独新增任务日志文件。

开始一个需求时，相关记录统一写入 `docs/WORKING_CONTEXT.md`，至少包含：
- 需求编号
- 需求名称
- 需求目标
- 需求范围
- 已知约束
- 当前状态
- 首条记录（需求确认）

开始写入前，必须先判断本次任务属于以下哪类，并据此决定是否更新全局状态区：
- 全局主线需求切换
- 当前主线需求下的子任务
- 已有需求的后续追加

开始实现前，至少补充：
- 已确认方案
- 计划改动点
- 预计涉及文件
- 如果本次属于全局主线需求切换，再更新全局 `当前下一步`；如果属于主线下子任务或后续追加，则保持全局状态区不变
- 一条“开始实现”记录

完成一个需求时，至少更新：
- 实际完成内容
- 实际改动文件
- 验证结果
- 遗留问题
- 当前状态
- 一条“完成”记录

如果后续对同一需求有更改或追加，必须继续在原需求记录下追加，至少包含：
- 本次追加原因
- 本次改动内容
- 影响范围
- 验证结果
- 当前状态
- 一条新的更新记录（写在 `记录` 中，不能脱离原需求另写）

`docs/WORKING_CONTEXT.md` 中每条需求记录必须使用统一模板：
- 需求编号
- 需求名称
- 需求目标
- 需求范围
- 已知约束
- 当前状态
- 已确认方案
- 计划改动点
- 预计涉及文件
- 实际改动文件
- 验证结果
- 遗留问题
- 记录（持续包含需求确认、开始实现、完成、后续追加等节点）

其中字段含义补充说明：
- `实际改动文件` 默认记录该需求截至当前节点的累计改动文件，不应在后续更新时只保留最近一次局部改动而丢失既有记录

## 任务完成规则
一个明确任务完成后，应先更新相关文档，再决定是否提交 Git。

默认要求：
- 更新 `docs/WORKING_CONTEXT.md`，记录当前结果、遗留问题和当前下一步
- 如果本次任务修改了协作方式或项目规则，同时更新相关规范文档
- 有明确产出时，文档和代码应尽量一起提交到 Git
- 如果当前线程只有讨论，没有形成实际产出，可以只更新文档，不强制提交 Git

## 包管理与初始化命令
项目默认使用 `npm` 作为包管理工具。

项目默认脚手架初始化命令为：
```bash
npm create vue@latest
```

## 项目初始化最小范围
项目初始化阶段默认只做到以下范围：
- 使用 Vue 3 + TypeScript 创建项目基础脚手架
- 接入 Vue Router、Pinia、Element Plus、Tailwind CSS、Axios
- 配置 ESLint 和 Prettier
- 建立基础目录结构、路径别名和全局样式入口
- 建立基础路由、基础布局和请求封装骨架

项目初始化阶段默认不包含以下内容：
- 具体业务页面实现
- 真实接口联调
- 权限体系完整实现
- 多语言、主题系统、复杂工程优化
- 单元测试、E2E、CI/CD、Docker

## 初始化后首批基础文件范围
项目初始化完成后，首批基础文件默认至少包含：
- `src/main.ts`
- `src/App.vue`
- `src/router/index.ts`
- `src/stores/index.ts`
- `src/api/request.ts`
- `src/layouts/AppLayout.vue`
- `src/views/dashboard/index.vue`
- `src/styles/index.css`
- `src/types/`
- `src/utils/`

## 目录结构约定
项目初始化后默认采用以下目录结构：

```text
src/
  api/
    modules/
  assets/
  components/
  composables/
  constants/
  layouts/
  router/
    modules/
  stores/
    modules/
  styles/
  types/
  utils/
  views/
    dashboard/
```

目录职责：
- `api/`：接口请求封装和接口模块
- `api/modules/`：按业务域拆分的接口文件
- `components/`：可复用通用组件
- `composables/`：组合式逻辑
- `layouts/`：全局布局组件
- `router/`：路由入口和路由模块
- `stores/`：Pinia 状态管理入口
- `stores/modules/`：按模块拆分的 store
- `styles/`：全局样式、变量和 Tailwind 入口
- `types/`：全局共享类型
- `utils/`：纯工具函数
- `views/`：页面级组件

## 命名规则
默认命名规则如下：
- Vue 组件文件和布局文件使用 `PascalCase`，例如 `AppLayout.vue`、`PageHeader.vue`
- 页面目录使用 `kebab-case`，页面入口统一使用 `index.vue`，例如 `views/user-management/index.vue`
- `composables` 使用 `useXxx.ts`，例如 `useTable.ts`
- `stores/modules` 使用 `xxx.ts` 文件名，导出统一使用 `useXxxStore`
- `api/modules` 使用 `xxx.ts` 文件名，按业务域组织接口
- `router/modules` 使用 `xxx.ts` 文件名，按业务域组织路由
- 普通 TypeScript 文件默认使用 `kebab-case`
- 常量文件可使用 `UPPER_SNAKE_CASE` 常量名，文件名仍使用 `kebab-case`
- 路由路径使用 `kebab-case`
- 路由 `name` 使用 `PascalCase`

## 工程规范
- 保持代码风格一致
- 默认遵循 ESLint 和 Prettier 约定
- 新增规则和长期约定应同步更新文档
- Element Plus 样式默认按三层分工维护：`src/styles/element-theme.css` 仅承载主题 token，`src/styles/element-overrides.css` 仅承载通用组件覆盖，页面/布局文件只承载局部结构样式
- 能通过 Element Plus 主题 token 或通用覆盖层解决的样式，不应继续散落在页面或布局组件内实现
- `:deep` 仅用于当前页面或当前布局独有的结构定制；如果同类 `:deep` 规则在第二处重复出现，必须抽离到统一覆盖层
- 不要在组件内直接通过 Element Plus 颜色属性、或大面积 `!bg-*`、`!text-*`、`!border-*` 去替代全局主题机制；只有明确的局部视觉例外才允许保留

## 禁止事项
- 不要把聊天记录当作长期上下文
- 不要把建议直接当成最终决定
- 不要在没有记录背景的情况下推进复杂任务
